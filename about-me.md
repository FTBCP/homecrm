\# about-me.md

> AI agents read this to understand how to work with me. Follow these preferences in every session.



\---



\## Who I Am



\*\*Name:\*\* Geoff

\*\*Role:\*\* Founder — I'm the product manager, designer, and business owner. I make all decisions.

\*\*Location:\*\* Chester County, PA (Philadelphia area)

\*\*Coding comfort:\*\* Non-technical. I do not write or read code. I use AI tools to build everything. When something breaks, I can paste errors and follow fix instructions — but every instruction must be spelled out step by step in plain English with exact commands to run or buttons to click. Never assume I know what a technical term means.

\*\*Computer:\*\* Windows 11

\*\*Primary IDE:\*\* Antigravity (may also use VS Code)

\*\*Terminal:\*\* Windows Command Prompt (not PowerShell, not bash)



\## How I Work



\*\*Instructions format:\*\* Give me instructions one line at a time. Each step should be a single action — one command to run, one button to click, one file to open. Number every step. If a step involves the terminal, give me the exact command to paste. If a step involves navigating a website or app, tell me exactly where to click and what to type. Never combine multiple actions into one step.



\*\*Technical language:\*\* Explain everything in plain English. If you must use a technical term, define it in parentheses the first time. Example: "Run this in your terminal (the black window where you type commands)."



\*\*Big changes:\*\* Show me the plan first and wait for approval before executing. One sentence per step is fine. Don't write a novel.



\*\*When something is unclear:\*\* Make your best guess and move forward. Flag what you assumed so I can correct it if needed. Don't block on a question I might not know the answer to.



\*\*After making changes:\*\* List the files changed and a one-line summary of what each change does. Don't over-explain. I'll ask if I don't understand.



\*\*Testing:\*\* Write tests as we go. Every new feature should have at least basic tests before moving to the next feature. Explain what each test checks in plain English.



\*\*Commit style:\*\* Small, frequent commits with plain English messages. No conventional commits format needed.



\## What I Care About



\- Speed over perfection. Ship it, then fix it.

\- Clean UI that looks good on mobile. I have design taste — if something looks off I'll call it out.

\- Minimal dependencies. Don't install a library for something you can build in 20 lines.

\- I'm building multiple projects at once. Respect my time — don't make me re-explain context that's already in memory.md or AGENTS.md.



\## What Slows Me Down



\- Technical jargon without explanation.

\- Instructions that skip steps or assume I know what to do next.

\- Being asked questions I already answered in a previous message.

\- Mac/Linux commands on my Windows machine. Always give me Windows-compatible commands.

\- Multiple actions crammed into one step. One action per step, always.



\## Terminal Reminders



I'm on Windows Command Prompt. Always use:

\- `echo. > filename` not `touch filename`

\- `type filename` not `cat filename`

\- `del filename` not `rm filename`

\- `dir` not `ls`

\- Backslashes in paths: `src\\components\\` not `src/components/`
